## 概要 (Description)
このプロジェクト「Inception」は、Docker を使用して複数のサービスを仮想化することで、システム管理の知識を広げることを目的としています。Docker Compose を使用し、特定のルールに従って異なるサービス（Nginx、WordPress、MariaDB）で構成される小規模なインフラストラクチャをセットアップします。

## 手順 (Instructions)
1. **前提条件**: マシンに Docker と Docker Compose がインストールされていることを確認してください。
2. **インストール**:
   リポジトリをクローンし、ルートディレクトリに移動します。
3. **実行**:
   以下のコマンドを実行して、インフラストラクチャを構築および起動します：
   ```bash
   make
   ```
   （または `docker-compose -f srcs/docker-compose.yml up --build` を使用）
4. **アクセス**:
   Web ブラウザを開き、`https://kosakats.42.fr` にアクセスします。

## 概念 (Concepts)

### VM vs Docker
- **仮想マシン (VM)**: ハードウェアを仮想化します。ハイパーバイザー上で完全なオペレーティングシステム（カーネル + ユーザー空間）を実行します。VM は重く、起動に時間がかかりますが、強力な分離を提供します。
- **Docker**: オペレーティングシステムを仮想化します。コンテナはホストのカーネルを共有しますが、ユーザー空間は分離されています。軽量で、ほぼ瞬時に起動し、リソース効率が優れています。

### Secrets vs Env
- **Env (環境変数)**: コンテナに渡されるキーと値のペアです。便利ですが、ログや `docker inspect` で見えることが多いため、機密データ（パスワード、API キー）には安全ではありません。
- **Secrets**: 機密データを管理するための安全な方法です。Docker では、シークレットは転送中および保存中に暗号化され、必要な場合にのみコンテナ内のファイル（通常は `/run/secrets/`）としてマウントされるため、漏洩のリスクが軽減されます。

### Docker Network vs Host
- **Docker Network**: コンテナ用の分離されたネットワーク環境を作成します。コンテナは DNS 名を介して通信し、ホストに明示的にマップされたポートのみを公開します。これがデフォルトであり、安全なモードです。
- **Host**: コンテナはホストのネットワーク名前空間を直接共有します。ホストの IP とポートを使用します（例：コンテナのポート 80 はホストのポート 80）。これにより、ネットワークの分離がなくなります。

### Volume vs Bind Mount
- **Volume**: Docker によって管理され、ホストファイルシステムの専用領域（例：`/var/lib/docker/volumes/`）に保存されます。データの永続化に推奨される方法であり、バックアップが容易で、異なる OS 間で動作します。
- **Bind Mount**: ホストマシン上の特定のファイルまたはディレクトリをコンテナにマップします。ホストのディレクトリ構造に依存します。開発（例：ライブコード更新）には便利ですが、ボリュームよりも移植性が低くなります。
